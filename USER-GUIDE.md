# AIPM 使用指南

## 快速开始

### 登录系统

1. 打开浏览器访问 `http://localhost:5173`
2. 输入邮箱和密码登录（管理员账户：`byron@aipm.dev` / `test123456`）
3. 登录后自动进入仪表盘

### 首次使用建议顺序

> 系统的核心逻辑：**先建规范（SOP），再管提示词，然后用投喂包驱动 AI 干活，最后沉淀经验。**

```
① 设置（建项目、建小组）
  → ② SOP 库（导入/编写规范文档）
    → ③ 提示词库（创建生产指令）
      → ④ 投喂包（组装投喂内容，按阶段执行）
        → ⑤ 经验沉淀（记录问题，AI 总结，合并到约束条件）
```

## 系统设置（设置入口）

进入左侧菜单最底部的「设置」。

### 项目管理

**你需要先建一个项目，后续所有内容都挂在项目下。**

- 点击「创建项目」→ 填写项目名称和描述
- 一个项目 = 一个产品线（如"Dashboard"、"官网"）

### 小组管理

**两人制小组是核心协作单元。**

- 在项目下创建小组，指定需求架构师和实施工程师
- 每个小组独立推进，互不阻塞

### 用户管理

- 添加团队成员，分配角色（ADMIN / ARCHITECT / ENGINEER / DESIGNER）
- ADMIN 可管理所有内容，其他角色只能修改自己创建的资源

### AI 配置

- 配置 OpenAI API Key（用于经验沉淀模块的 AI 总结和仓库审核）
- 默认模型：gpt-5.4
- 可调整温度和最大 token 数

## 核心模块使用

### 1. SOP 库 — 先定规范

> SOP = 标准作业流程文档。告诉团队"做什么、怎么做、什么标准"。

**作用：** 存放项目的规范文档，作为 AI 执行任务时的上下文输入。

**操作：**
1. 点击「创建 SOP 项目」→ 填名称、选关联项目、设可见范围
2. 进入 SOP 项目 → 按八层结构添加文档：
   - **产品需求**：PRD、用户故事
   - **内容素材**：文案、图片说明
   - **设计系统**：颜色、字体、组件规范
   - **前端架构**：技术选型、目录结构、路由规划
   - **后端架构**：API 设计、数据库 Schema、服务分层
   - **AI 提示词**：各阶段的提示词引用
   - **验收标准**：测试用例、质量标准
   - **附录**：参考资料、术语表

**关键点：**
- SOP 文档支持 Markdown 格式
- 每次修改自动创建版本，可查看历史和 diff
- 可见范围：私有（仅自己）、团队（小组内）、公开（全公司）
- 只有创建者和管理员能修改/删除

---

### 2. 提示词库 — AI 的生产指令

> 提示词 = 给 AI 的任务指令。告诉 AI "用什么规范，生成什么结果"。

**作用：** 管理所有给 AI 的提示词，支持版本控制和社区协作。

**操作：**
1. 点击「创建提示词」→ 填名称、选分类、写提示词内容
2. 分类：设计 / 前端 / 后端 / 测试 / 集成 / 优化
3. 可指定「依赖的 SOP 层」— 告诉系统这个提示词需要哪些 SOP 文档作为上下文

**协作机制（类似 GitHub）：**
- **Star**：收藏好用的提示词
- **复制（Fork）**：复制别人的提示词到自己名下，自由修改
- **PR**：对别人的提示词提交修改建议，原作者审核合并
- **版本历史**：每次修改都有记录

**分类筛选 + 排序：**
- 按分类 Tab 切换（全部 / 设计 / 前端 / 后端 ...）
- 排序：最近更新 / 最多收藏 / 最多复制

---

### 3. Skill 库 — 辅助能力

> Skill = 可复用的辅助工具/能力。比如"PRD 完善助手"、"Code Review"。

**与提示词的区别：**
- 提示词 = 一次性的生产指令（"生成首页代码"）
- Skill = 持续可用的辅助能力（"帮我审查代码质量"）

**操作：**
1. 创建 Skill → 填名称、分类（阶段助手 / 质量检查 / 通用工具）
2. 可关联 Git 仓库地址（如果 Skill 有对应的开源实现）
3. 同样支持 Star、复制、搜索

---

### 4. 模板库 — 规范框架

> 模板 = 可套用的文档框架。比如"PRD 模板"、"验收标准模板"。

**与提示词的区别：**
- 提示词给 AI 用
- 模板给人用，是文档的骨架，套用后自己填内容

**操作：**
1. 创建模板 → 填名称、选分类、写模板内容（Markdown）
2. 使用时：复制模板到自己的项目，在框架基础上编辑
3. 支持 Star、复制、版本历史

---

### 5. 投喂包 — 驱动 AI 干活的核心

> 投喂包 = SOP + 提示词 + 项目上下文，自动组装后投喂给 AI。

**这是整个系统最核心的模块。**

**三层结构：**
```
公开层（Prompt）     → 提示词指令框架，可社区共享
核心层（Core）       → 约束规范（禁止清单、经验教训），真正的 IP
上下文层（Context）  → PRD、技术架构等项目级文档
```

**操作流程：**

1. **创建投喂包**
   - 选择关联的迭代（Iteration）
   - 填名称、选阶段（设计 / 实施）
   - 可关联一个提示词
   - 设置依赖关系（哪些投喂包必须先完成）

2. **添加文件**
   - 给投喂包添加核心层文件（约束条件 MD）和上下文层文件（PRD、架构文档）
   - 这些文件会和提示词一起组装成完整的投喂内容

3. **组装查看**
   - 点击「组装」可预览完整的投喂内容
   - 系统自动将三层内容拼接成一份完整的 Markdown

4. **执行记录**
   - 投喂给 AI 后，记录使用的 AI 工具、输出摘要、遇到的问题
   - 状态流转：阻塞 → 待处理 → 进行中 → 审核 → 完成 / 返工

5. **依赖图**
   - 切换到「依赖图」视图，可视化查看投喂包之间的依赖关系
   - 上游未完成的投喂包会阻塞下游

**典型使用场景：**
```
投喂包 B1（后端-项目初始化）
  → 投喂包 B2（后端-数据库设计）—— 依赖 B1
    → 投喂包 B3（后端-API 开发）—— 依赖 B2
投喂包 F1（前端-项目初始化）
  → 投喂包 F2（前端-页面开发）—— 依赖 F1 + B3
```

### 6. 协同白板 — 可视化编排

> 白板 = 把提示词卡片拖到画布上，可视化编排执行顺序。

**操作：**
1. 创建白板 → 取个名字
2. 右侧面板搜索/筛选提示词 → 点击添加到画布
3. 在画布上拖拽排列，规划执行顺序
4. 支持多人同时查看（WebSocket 实时同步光标）
5. 导出白板内容为投喂包编排

**适用场景：** 开迭代规划会时，大家一起在白板上挑选和排列本次迭代要用的提示词。

---

### 7. 设计管理 — 看板式管理

> 管理 AI 生成的设计稿的精修流程。

**状态看板：**
```
AI 生成 → 待精修 → 精修中 → 已确认 → 已锁定
```

**操作：**
1. 创建设计稿 → 填名称、关联迭代、粘贴 Figma 链接
2. 设计稿在看板上按状态分列展示
3. UI 设计师确认后锁定，锁定后不可修改

---

### 8. OKR 看板 — 目标管理

**操作：**
1. 选择项目 → 创建 Objective（目标）
2. 在目标下创建 Key Result（关键结果）
3. 手动更新 KR 进度
4. 看板展示目标达成情况

---

### 9. 经验沉淀 — 闭环改进

> 把踩过的坑变成约束条件，防止重复犯错。

**完整流程：**
```
① 提交反馈（描述问题）
  → ② AI 自动总结（分类、提炼、生成约束条件）
    → ③ 仓库 Agent 审核（查重、打标、质量评估）
      → ④ 人工审批合并（需求架构师最终确认）
```

**操作：**
1. 点击「提交反馈」→ 描述遇到的问题
2. 可关联具体的迭代和投喂包（支持搜索选择）
3. 提交后 AI 自动分析，生成结构化总结
4. 管理员在详情页审批：合并到约束条件库 / 驳回

**状态流转：**
- 已提交 → AI 已总结 → 待审核 → 已合并 / 已驳回

---

### 10. 仪表盘 — 全局总览

**三级下钻：**
- **公司级**：所有项目卡片，每个项目显示小组数量
- **项目级**：点击项目卡片 → 查看该项目下所有小组
- **小组级**：每个小组显示当前处于哪个步骤（①定规范 → ⑤验收）

可按月份筛选查看历史数据。

## CLI 命令行工具

在终端中使用，适合开发者快速操作。

```bash
# 登录（获取 token）
aipm login

# 拉取投喂包到本地
aipm pull --iteration <迭代ID> --output ./feeds/

# 列出资源
aipm list prompts              # 列出提示词
aipm list skills               # 列出 Skill
aipm list feeds                # 列出投喂包
aipm list templates            # 列出模板

# 查看状态
aipm status

# 提交经验反馈
aipm feedback --description "xxx问题" --iteration <迭代ID> --feed <投喂包ID>
```

## 典型工作流程示例

### 场景：启动一个新产品模块的开发

```
1. 【管理员】设置 → 创建项目"用户中心" → 创建小组（张三+李四）

2. 【需求架构师-张三】
   a. SOP 库 → 创建 SOP 项目 → 导入/编写八层文档
   b. 提示词库 → 创建各阶段提示词（B1初始化、B2数据库、F1前端初始化...）
   c. 投喂包 → 创建投喂包，关联提示词，设置依赖关系
   d. 白板 → 拖拽排列本迭代的执行顺序
   e. 投喂包 → 组装内容 → 复制投喂给 AI（Cursor/Claude）

3. 【实施工程师-李四】
   a. CLI → aipm pull 拉取投喂包
   b. 按顺序执行投喂包，记录执行结果
   c. 遇到问题 → 经验沉淀 → 提交反馈

4. 【需求架构师-张三】
   a. 经验沉淀 → 审核 AI 总结 → 合并有价值的约束条件
   b. 约束条件自动进入下次投喂包的核心层 → 形成改进闭环

5. 【管理员】
   a. 仪表盘 → 查看各小组进度
   b. OKR 看板 → 跟踪目标达成情况
```

## 常见问题

**Q：SOP 库和提示词库有什么区别？**
- SOP 库 = 规范文档（给人看、给 AI 当上下文）
- 提示词库 = 执行指令（直接投喂给 AI 让它干活）
- 投喂包把两者组装在一起

**Q：Skill 和提示词有什么区别？**
- 提示词 = 一次性任务指令（"生成登录页代码"）
- Skill = 持续可用的能力（"代码审查助手"，可关联 Git 仓库）

**Q：模板和 SOP 有什么区别？**
- 模板 = 空框架，套用后自己填
- SOP = 已填好的完整规范文档

**Q：别人能改我上传的公开资源吗？**
- 不能。只有创建者和管理员能修改/删除。其他人只能查看、收藏、复制。
